Delivered Ingestion

Design and build robust data ingestion pipelines to incorporate, normalise and continuously refresh the third-party data sources that power our platform.

What was built

  • ▸Connector framework with a shared base providing typed errors, token-bucket rate limiting, circuit breaker (closed / open / half-open), structured logs, Prometheus metrics.
  • ▸Four real adapters: Yahoo Finance (active), Binance, CoinGecko, CryptoPanic. Each translates raw HTTP into pydantic-typed records.
  • ▸Idempotent UPSERTs on natural keys: INSERT ... ON CONFLICT DO UPDATE. Replays over overlapping windows are safe.
  • ▸Continuous refresh via Celery Beat (1m / 5m / 1h cadences) — quotes for 18 instruments refresh every minute.

Delivered Schema design

Own data modelling and schema design across our platform, making sure the data layer is structured to deliver performance, flexibility and long-term scalability.

What was built

  • ▸SCD Type 2 on the instrument dimension with a partial unique index (WHERE is_current = TRUE) — supported natively by Postgres, no triggers needed.
  • ▸Declarative range partitioning on fact_price_candle by month. 10 partitions covering Oct 2025 → Jul 2026, auto-extended by a Celery maintenance task.
  • ▸Index strategy: BRIN on ts (~100× cheaper than B-tree for append-only), composite B-tree on (instrument_id, ts DESC) for last-N queries.
  • ▸Medallion layout: Bronze (raw JSONL.gz on MinIO), Silver (normalized in Postgres), Gold (materialized views refreshed every 5 min).

Delivered Infra · orchestration · warehousing

Own the data infrastructure tooling, including pipeline orchestration, storage and warehousing, together with development of the application layer.

What was built

  • ▸Dagster for asset-based orchestration: bronze → silver → gold lineage, sensors, schedules, partitioned assets. Heavy parallelism delegates to Celery via group().
  • ▸MinIO (S3-compatible) for raw bronze: JSONL.gz partitioned by source/date/hour, immutable, replayable.
  • ▸PostgreSQL for OLTP serving + silver. DuckDB embedded as an analytical layer that scans Postgres via postgres_scanner — analysts run heavy queries without touching the API path.
  • ▸Kubernetes manifests with kustomize overlays (staging / production), HPA, PDB, NetworkPolicy, migrations Job, ingress with cert-manager.
  • ▸CI/CD with GitHub Actions (lint + tests + Docker build + GHCR push), CodeQL security scans, Dependabot grouping.

Architectural — demonstrated via ADRs Strategy

Collaborate directly with the CTO to define data strategy and technical architecture as the platform scales.

Honest framing

There's no real CTO here, but every consequential decision is captured in a written Architecture Decision Record — exactly what a CTO review session produces:

  • ▸ADR 0001 Django vs FastAPI — chose batteries-included, accepted the async limitation
  • ▸ADR 0002 Celery vs RQ vs arq — chose Celery for routing + retry semantics
  • ▸ADR 0003 Dagster vs Airflow vs Prefect — chose Dagster for asset model
  • ▸ADR 0004 Medallion architecture — bronze immutability for replays
  • ▸ADR 0005 Postgres declarative partitioning — chose stock Postgres over Timescale / Citus
  • ▸ADR 0006 DuckDB analytical layer — isolation from OLTP
  • ▸ADR 0007 Structured logging — JSON + request-id propagation

What's in each ADR

  • ▸Context — why the decision is needed
  • ▸Decision — what we chose, briefly
  • ▸Rationale — why over the alternatives
  • ▸Trade-offs — what we gave up. Honest about limitations.
  • ▸Alternatives considered — and why each was ruled out
